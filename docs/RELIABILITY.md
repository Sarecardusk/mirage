# Mirage Reliability Boundaries

> Last Updated: 2026-04-05

本文件定义数据与流程可靠性的硬约束，目标是防止“可用但不可恢复”的实现进入主线。

## 存储与作用域不变量

- 主存储固定为 `SurrealDB 3.x (SurrealKV)`。
- 默认隔离模型固定为 `单用户单工作区 + 同 Theme Card 下多 Session`。
- 会话态对象必须携带 `session_id` 与 `theme_card_id`。
- `Profile` 仅作为保留扩展维度，不作为会话隔离必需字段。

## Session 隔离不变量

- 同一 `Theme Card` 下不同 `Session` 的消息历史必须独立（MH-04）。
- 同一 `Theme Card` 下不同 `Session` 的检索记忆上下文必须独立（MH-05）。
- 卡片共享设定（如角色设定、共享指令）在该卡片全部 Session 间复用（MH-06）。
- 新建 Session 默认不继承旧 Session 的隐式记忆，仅允许显式克隆路径。

## Theme Card 演进约束

- 每个 `Theme Card` 必带 `schemaVersion`（MH-02）。
- 读取旧版本数据时必须走显式迁移管线（MH-03）。
- 迁移失败必须返回可追踪错误，禁止 silent fallback（HR-05）。

## 二进制资产一致性

- 使用 `SurrealDB bucket storage` 存储二进制资产。
- `Theme Card` 仅持有资产引用 ID，不持有资产副本。
- 删除资产前必须校验引用关系，禁止产生悬挂引用（HR-06）。

## 关键流程失败路径

### 创建并保存 Theme Card

- 校验失败：返回字段级可读错误并标注来源（Zod 或 validator）。
- 元数据写入失败：回滚并返回可重试错误（MH-12）。
- 资产写入失败：不落悬挂引用，返回资源错误码（MH-13）。

### 检索 + 推理

- 检索失败：允许降级为“无检索上下文”，并发出降级事件。
- 提供方失败：返回标准化错误，包含 `retryable` 标识（MH-10）。
- 归一化失败：视为网关内部错误，不向 UI 透出供应商原始结构（MH-11）。

## 应用级迁移策略

- SurrealDB 中维护 `app_schema_version` 元数据记录。
- 启动时先执行应用级迁移（数据库结构变更），再执行实体级迁移（`ThemeCard` schemaVersion）。
- 应用级迁移失败必须阻塞启动并返回可追踪错误，不允许带着旧 schema 运行新版本应用（MH-24）。

### 部分迁移恢复策略

- 每次迁移步骤应在 SurrealDB 事务内执行。事务失败时由 SurrealDB 自动回滚，不留中间态。
- 若迁移涉及多步且无法在单一事务内完成，每步须记录已完成的步骤编号到 `migration_progress` 元数据。重启后从上次成功步骤继续（幂等重入）。
- 迁移函数必须满足幂等性：对同一数据重复执行不产生副作用，允许安全重试。

### 迁移实现规则

- **顺序迁移**：仅支持 N → N+1 逐版本迁移，不支持跳版本。读取 schemaVersion 为 1 的数据时，需依次执行 1→2、2→3……直到当前版本。
- **函数放置**：迁移函数定义在 `infra/` 层（因为需要访问存储），版本常量定义在 `domain/` 层。
- **函数签名**：每个迁移函数接收旧版 JSON 值，返回新版 JSON 值（`fn migrate_v1_to_v2(old: serde_json::Value) -> Result<serde_json::Value, InfraError>`）。
- **注册**：迁移函数按版本号顺序注册到迁移管线中，管线在启动序列步骤 2 和实体读取时自动调用。

## 验收场景

- 旧版 `Theme Card` 升级后字段完整可读。
- 迁移失败可定位、可追踪、可复现。
- 资源删除后不存在悬挂引用。
- 检索失败与模型失败都能走可展示降级路径。
- 同一卡片下不同 Session 间不存在记忆串扰。
- 会话级配置变更不污染同一卡片其他 Session。
- 同步占位不影响单机流程。

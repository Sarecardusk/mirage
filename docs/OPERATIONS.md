# Mirage Operations And Diagnostics

> Last Updated: 2026-04-05

本文件定义 Mirage 的最小可观测性与诊断约束。
目标是让问题可定位，而不是把日志当成调试噪音堆积。

## 最小诊断字段

所有关键命令与失败路径的日志至少包含：

- `correlation_id`
- `command_name` 或 `event_name`
- `theme_card_id` / `session_id` / 相关实体 ID
- `duration_ms`
- `result`（success / error / cancelled / degraded）
- `error_code`（失败时）

## 关联规则

- `correlation_id` 由 Rust 命令层生成。
- 同一次命令调用产生的日志、RuntimeEvent、结构化错误必须共享同一个 `correlation_id`。
- 前端若再次发起显式重试，应生成新的调用链；不得把不同调用混成同一个关联 ID。

## 脱敏规则

以下内容禁止进入普通日志、事件 payload、错误 message：

- API Key、口令、派生密钥、签名原文。
- 未脱敏的用户敏感输入。
- 原始供应商响应全文。
- 可直接重放的认证头与令牌。

允许记录的是：

- 脱敏后的摘要。
- 字段缺失、校验失败、状态码、错误码。
- 诊断所需的长度、类型、阶段、耗时。

### 脱敏操作定义

"脱敏后的摘要"指以下操作之一，不得超出此范围：

- **存在性**：仅输出布尔值（如 `api_key_present: true`），不输出值本身。
- **长度与类型**：仅输出字段长度或类型标签（如 `input_length: 42`），不输出内容。
- **末位掩码**：仅保留末 4 字符，其余替换为 `*`（如 `****ab12`）；适用于需要人工对照确认身份的场景。
- **哈希指纹**：输出值的 SHA-256 前 8 字符（如 `fingerprint: a1b2c3d4`）；适用于关联排查但不暴露原文的场景。

禁止输出：key 前缀（如 `sk-`）、token 类型、部分明文片段、Base64 编码后的值。

## 事件与日志分工

- 日志用于开发者定位问题。
- `RuntimeEvent` 用于 UI 展示状态、进度、告警与可恢复提示。
- 同一事实若同时写日志和发事件，字段名应保持可对照，不与 UI 文案耦合。

## 预算与告警

- 启动到 `AppReady` 的参考预算仍按 `DESIGN.md` 性能预算章节执行（启动 < 3s）。
- 非 LLM 命令超过参考预算（< 100ms，见 `DESIGN.md`）时，至少记录慢命令警告日志。
- 背压、取消、迁移失败、安全硬拒绝都必须留下可定位日志。

## 当前阶段要求

- 在还没有专门诊断导出功能前，结构化日志就是最小诊断资产。
- 任何新增高风险流程，如果无法回答“失败后怎么定位”，视为约束未完成。
